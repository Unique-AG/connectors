Topic: Overall KYC/KYB Flow with KICR (Kicker) APIs

KICR provides a unified, normalized API layer over 200+ national company registries, removing the need for clients to build and maintain individual integrations. The APIs are intentionally simple (mostly GET endpoints, with a single POST for document ordering) and return consistent JSON structures across jurisdictions, despite large differences in registry quality, structure, and latency. KICR’s core value is real‑time, registry‑sourced data suitable for high‑regulation use cases (e.g., banking AML/KYB), where up‑to‑date, authoritative information is required. The typical KYC/KYB flow is: search for a company, obtain the KICR “kicker ID,” then retrieve detailed profiles and/or documents as needed.

Topic: Search API – Entry Point and Usage


The Search API is the mandatory first step and is free. It is used to verify that a company exists and is active before any paid calls, and to obtain the KICR “kicker ID” (unique company identifier) required for subsequent API requests. The caller must specify the company’s country; a global search exists for unknown countries but the preferred approach is country‑specific queries. Search can be by company name or company number: name search returns multiple results and is less suitable for full automation, while number search usually yields a single precise match and is recommended for automated flows. The UK is used as a demo jurisdiction because it exposes almost all possible fields; other registries, especially outside the EU/Australia/Singapore/Hong Kong, may return minimal data (sometimes only “company exists and is active”). The search response includes basic details (address, incorporation date, status) and the KICR ID at the top. In automated flows, the system typically just confirms the correct company and active status, then proceeds.

Topic: Enhanced Profile API – Structure, Content, and Normalization

Once the KICR ID is known, the Enhanced Profile API returns detailed company information, including directors and shareholders where available. The response includes:

Metadata: Contains pricing information (e.g., a UK enhanced profile costs 1 KICR credit).

Activity: Uses standardized NACE and SIC codes, allowing clients to reuse existing mappings.

Address: Always includes a single full address string to ensure completeness, since many registries lack structured address fields. Structured components (street, city, postcode) are provided where possible but not guaranteed.

Shareholding: Indicates whether each shareholder is a person or a company. Company shareholders signal a UBO chain that may require further investigation; person shareholders represent UBOs at that level. Registries generally do not provide shareholder country, so this is absent.

Directors: Lists directors with name, whether they are a person, address, appointment date, and role; multiple directors can be present.

Persons of Significant Control (PSC, UK‑specific): For the UK, KICR includes PSC data from Companies House because it is often requested. However, PSC entries are self‑declared and frequently do not reflect true UBOs, so they should not be treated as authoritative UBO information.

Source and Normalization: The response identifies the registry source (e.g., Companies House for the UK). Dates are provided both in original registry format and in a normalized format to handle differing conventions (e.g., US vs UK formats, or raw numeric strings like in the Netherlands). Legal statuses are mapped from over 1,000 registry‑specific values into a normalized set (e.g., active, inactive, distressed), saving clients from building and maintaining their own mapping logic.

PDF Option: For evidencing and audit, KICR can provide a PDF with a registry stamp. Banks often prefer such “official” documents over raw JSON for compliance and audit trails.

Topic: Light Profile – When Enhanced Profiles Are Not Available

In some jurisdictions, KICR cannot provide full enhanced profiles with directors and shareholders. Reasons include directors/shareholders only being available in PDFs (sometimes handwritten, making OCR unreliable) or registries (e.g., in the US) not exposing directors in a way suitable for real‑time APIs. For these cases, KICR offers a Light Profile:

Structurally similar to the enhanced profile but without directors and shareholders.

Client integration must implement per‑country logic to decide whether to:

Request an enhanced profile (if available),

Request a light profile (if enhanced is not available), or

Order specific documents (PDFs) to obtain directors/shareholders.

KICR can provide a spreadsheet listing all countries and indicating whether an enhanced profile is available. This mapping is relatively stable because registry behavior changes slowly.

Topic: Document Ordering API – Listing, Selecting, and Retrieving PDFs

Where directors and shareholders are only available via registry documents (PDFs), KICR exposes APIs to list and order these documents.

List Available Documents: Using the KICR ID, the documents endpoint returns all available documents for a company. Each entry includes document name (e.g., “shareholders document”, “extended profile”), cost in credits, expected delivery time, and a document ID. KICR’s documentation explains, per jurisdiction, which document types contain directors/shareholders.

Select the Right Document: In automated flows, an AI agent or rules engine uses the country and KICR documentation to identify the correct document type, optionally filtering by keywords in the document name (e.g., “shareholders”, “extended profile”).

Order the Document (POST): This is the only POST endpoint. The request includes the KICR company ID, the document ID, and an optional internal customer reference for billing/traceability. The response confirms the order and returns an order reference with status “pending”.

Polling for Completion (No Webhooks): KICR does not yet support webhooks. Clients must poll an endpoint to check order status. Status values include “pending” and “success”; on “success” a download link for the PDF is provided. High‑volume users may poll every minute and automatically download all newly completed orders; lower‑volume users may poll less frequently (e.g., every 10 minutes). The polling endpoint can be filtered by date range to avoid retrieving the full order history each time.

Delivery Times and Manual Intervention: Documents with “5 minutes” delivery time are delivered via API and typically arrive within that window. Longer times (e.g., 15 minutes, 1 hour) indicate manual steps either at KICR or at the registry (e.g., physical retrieval and scanning from a filing cabinet in jurisdictions like BVI). Delivery times are expected SLAs, not hard guarantees, but KICR reports that 99% of documents are delivered within or under the indicated time.

Topic: Registry Coverage, Status, and Downtime Handling

KICR maintains integrations with many registries worldwide, ranging from modern, well‑funded systems to under‑funded, legacy registries with slow or manual processes. KICR provides a public status page showing availability per registry (e.g., “Romania registry down for two hours”). The API will also indicate “registry unavailable” when a registry is down.

Current limitations and roadmap:

There is no API to query the status page directly yet.

Clients can subscribe to the status page (e.g., via email) and have an AI agent monitor it.

KICR plans to add an API endpoint to query coverage and capabilities per country (e.g., “for Switzerland, what can I buy?”), but this is not yet available.

Historically, the product targeted human users (bank/insurer analysts) and is now being adapted for more automated, large‑scale use.

Integration implications:

Clients must design for edge cases such as registry downtime (queue and retry later, or handle failures gracefully) and incorrect input from end customers (wrong company number, etc.).

These issues would also exist if connecting directly to registries; KICR cannot control registry uptime.

Topic: Sandbox / Test Environment and Its Limitations

Because registries generally do not provide sandbox environments, KICR’s sandbox is simulated:

It uses a storage blob with ~700 companies representing “good examples” for each jurisdiction.

Name search in sandbox: Returns all companies in that country without real search logic; useful for testing request/response structure but not realistic name matching.

Number search in sandbox: Works correctly and is the recommended way to test precise lookups.

The sandbox is not suitable for large‑scale testing (e.g., 500+ companies) and is intended mainly to validate integration logic and understand response structures.

For more realistic testing, once a client commits to integration, KICR can provide complimentary production credits to test against live registries before going live.

Topic: Pricing, Credits, and Cost Considerations

KICR uses a credit‑based pricing model. Each enhanced profile, light profile, or document has a cost in KICR credits, indicated in the metadata of API responses. For example, a UK enhanced profile costs 1 credit, while some documents in jurisdictions like BVI can cost around 40 EUR. This has implications:

Bulk ingestion of profiles to build a large internal database or vector store is very expensive, especially in high‑cost jurisdictions.

The service is designed for targeted, on‑demand retrieval at the point in the process where the client is ready to confirm directors/shareholders, not for mass data harvesting.

Typical KYC/KYB usage: use free search to identify the company, then purchase only the specific profile or document needed for that case.

Topic: Tracking Usage per End Customer (Reseller Scenario)

For resellers who serve multiple end customers, KICR supports cost attribution via a customer reference parameter:

KICR issues a single production API key per contracting customer (the reseller).

The reseller is billed by KICR, not the end customers.

Each purchase request (enhanced profile, light profile, document order) can include a customer reference parameter encoding the end customer (e.g., “ABC Bank”) and an internal transaction number.

When retrieving the list of all orders via the relevant endpoint, each order includes this reference, enabling attribution of costs to specific end customers and reconciliation of billing and usage.

The exact parameter name is documented in KICR’s API documentation (the presenter did not recall it during the call).

10. Topic: API Keys, Tenancy Models, and Multi‑Tenant Considerations

KICR’s current model provides one production API key per customer, with no built‑in multi‑tenant key management or delegated permission model. Implications:

The client operates in multiple deployment models:

Multi‑tenant SaaS, where multiple clients share the same platform.

Customer‑managed deployments, where software runs in the customer’s own tenant.

In customer‑managed deployments, the end customer could contract directly with KICR and use their own API key.

In the multi‑tenant SaaS model, the platform uses a single KICR key and relies on the customer reference parameter to track usage per end customer.

Key rotation or per‑tenant key strategies must be implemented by the client if required; KICR does not provide this natively.

11. Topic: MCP (Model Context Protocol) and AI‑Specific Integration

Florian asked about KICR’s plans regarding Anthropic’s Model Context Protocol (MCP) or similar AI‑oriented standards that expose tools and prompts directly to LLMs. KICR’s current stance:

The roadmap is dominated by maintaining and upgrading registry integrations, improving coverage and real‑time access, and adding automation around coverage reports and similar features.

A new CTO with a strong AI focus has joined, so AI‑related enhancements are expected in the future.

There is currently no committed plan for MCP‑style LLM integration or a dedicated AI tooling layer.

For now, any MCP or LLM‑tooling integration must be built by the client as a wrapper around KICR’s REST APIs.

12. Topic: Real‑Time vs Cached Data and Suitability for AI Pipelines

KICR is designed around real‑time access to registries:

Profiles are not cached; each request goes to the registry in real time.

The only caching mentioned is for the UBO product, where a purchased UBO report is cached for 24 hours to avoid double‑charging if the same report is requested again.

This real‑time approach differentiates KICR from competitors that rely on stored, potentially outdated data.

Implications for AI/LLM architectures:

Typical AI pipelines involve periodic data ingestion, chunking, vectorization, and building a knowledge base for semantic search.

With KICR’s model, bulk ingestion of all companies is not feasible (costly and often unsupported by registries), and bulk registry data is often out of date, conflicting with KICR’s real‑time value proposition.

Recommended usage: treat KICR as the final, authoritative step in a KYC/KYB process once the target company is known and needs verification, rather than as a bulk data source for a static knowledge base.

Clients can periodically fetch data for specific companies and ingest it into a vector store, but this must be carefully scoped to control cost and remain compliant with regulatory requirements for up‑to‑date data (especially for banks).

13. Topic: Historical and Inactive Companies

Elena asked about how long registries retain information on inactive companies. KICR’s understanding:

In principle, registries do not delete company records once registered.

Historical data is retained to support use cases such as:

Tracking directors who appear repeatedly across multiple companies.

Identifying patterns of multiple failed or inactive companies associated with the same director, which can be a red flag for banks.

Example (UK): a company profile lists a director’s previous directorships. If a director has been involved in many companies that became inactive in a short period, this can indicate AML/KYB risk.

While KICR cannot guarantee identical behavior across all registries, the general expectation is that inactive companies remain accessible.
